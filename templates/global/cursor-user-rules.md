Understudy — Global Team (paste into Cursor → Settings → Rules → User Rules)

Cursor has no scriptable global rules directory today, so this file is
generated by `understudy --global` for you to paste manually, once, into
Cursor's User Rules field (Settings → Rules → User Rules). Re-run
`understudy --global` anytime to regenerate it after changing models,
guardrails mode or roles — then paste it again to refresh.

This applies to every workspace opened in Cursor on this machine. A
repository's own `.cursor/rules/` (deployed by `understudy` or
`understudy --here`) still applies for that repo on top of this.

## Team context

This machine uses the Understudy system: a team of specialized AI agents.

Available roles:
- architect — Solution design and architectural decisions
- backend — API, service and business logic implementation
- frontend — User interfaces and experience
- devops — Infrastructure, CI/CD and operations
- security — Security integrated throughout the cycle
- qa-engineer — Testing and software quality

## Spec-Driven Development (per repository)

- If the current repository has docs/spec.md, treat it as the living
  contract: read it and docs/decisions.md / docs/session-log.md before
  working, and update docs/session-log.md at the end of the session.
- If it does not, this repository has not been localized yet. You may still
  work directly, or offer to run `understudy --docs-only --yes` yourself via
  the terminal to add persistent spec/ADR/session-log tracking for that
  specific repository — the team itself keeps coming from this global
  install.

## Turning this repo into a persistent project

If the user expresses — in any words — that this repository should have
real, persistent memory rather than relying on session-by-session context
alone (e.g. "let's set this up properly", "guarda esto como proyecto",
"quiero spec-driven aquí"), offer to run `understudy --docs-only --yes`
yourself via the terminal tool and report what it created. There is no
slash command for this in Cursor (no global commands directory), so running
the CLI directly is the way to do it here. It creates `docs/spec.md`,
`docs/decisions.md`, `docs/session-log.md` and `docs/team-roster.md` for
this specific repo, without touching source code or overwriting anything
that already exists — it does **not** deploy project-specific rules; the
team keeps coming from this global install. If the user instead wants
per-project customization (different models per role, project-specific
rules instead of these shared global ones), run `understudy --here --yes`
instead.

## Recognizing good stopping points

A long conversation costs more (tokens, time, money) than starting fresh
with a concise summary. Watch for natural stopping points — a feature just
finished with tests passing, a bug fixed and verified, a question fully
answered — and proactively suggest wrapping up: if this repo has
docs/session-log.md, update it with what was done, what's pending and any
decisions made (there's no slash command for this in Cursor, just do it
directly); then tell the user this is a good point to close this chat and
start a new one, since the next session picks up context from
docs/session-log.md (if this repo has one) or this global team.

Don't interrupt in-progress work to suggest this, and never refuse to keep
going if the user wants to continue — it's a suggestion at natural breaks,
not a requirement.

## Code standards

- Readable and maintainable code for any team member
- Single-responsibility functions
- Business domain names, not generic names
- Explicit error handling with context in messages
- No hardcoded secrets — use vault/env vars
- No dead code or TODO comments in commits
